A B-tree index is a balanced, sorted tree stored in fixed-size database pages: a small root page points to intermediate branch pages, branch pages contain separator keys and child-page pointers, and leaf pages contain keys in sorted order plus either row locations, primary-key values, or—in a clustered index—the rows themselves; databases usually use the B+tree variation, where complete index entries live at the leaves, leaves are linked left-to-right for range scans, internal keys merely guide navigation, and every leaf remains at the same depth, so even a huge index normally needs only a few page visits.

To look up a value, the database reads the root, compares the search key with its separators—often using binary search within the page—follows the corresponding child pointer, repeats through any intermediate levels, and searches the chosen leaf; equality lookup therefore costs roughly one page read per tree level, often fewer physical reads because upper pages are cached, while a range lookup descends once to the first matching leaf and then walks linked leaf pages sequentially until the range ends, and composite indexes are most useful when the query constrains their leftmost columns because entries are ordered lexicographically, such as `(customer_id, date)` first by customer and then by date.

On insertion, the database descends to the correct leaf and places the new entry in order; if enough free space exists, only that page and its logging metadata change, but if the page is full, the engine allocates another page, redistributes entries between the old and new pages, and inserts a separator plus pointer into the parent, which can overflow and split recursively up the tree, with a root split creating a new root and increasing the height by one, while deletion usually marks or removes an entry and may merge or redistribute sparse pages—although implementations often postpone merging to avoid churn—and monotonically increasing keys tend to concentrate inserts at the right edge whereas random keys spread writes but cause more scattered page access, fragmentation, and mid-tree splits.

The central trade-off is that an index spends storage, memory, and write work to make reads faster: it can turn a full-table scan into a few page visits, support ordered retrieval, ranges, joins, uniqueness checks, and sometimes answer a query entirely from included columns, but every insert, delete, or indexed-column update must also modify each affected index, generate transaction-log and replication traffic, acquire latches or locks, and occasionally split pages, so extra or wider indexes improve selected reads while reducing write throughput and cache efficiency; page fill-factor settings can reserve room and reduce future splits at the cost of larger, less densely packed indexes and more read pages, and the exact balance depends on workload, selectivity, caching, concurrency, storage, and engine behavior—especially because a low-selectivity query may still favor a table scan even when a B-tree index exists.
