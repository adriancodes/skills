# Running the Create Skill Eval

0. Run `node check-workflow-contract.mjs`. This zero-token regression check must pass before any actor session.
1. Preserve every opportunity-discovery run under `results/opportunity/`. For each discovery case × `none|prompt` arm, add one typed `suite-manifest.json` entry containing case/arm, hashed raw output, hashed tool trace, hashed artifacts, scored assertions, tokens, and actor sessions. Every discovery assertion uses the blinded judge, includes hashed judge prompt/output, and equals the judge output. Record an `opportunity_decision` of `PROTOTYPE` with residual assertion IDs that actually failed in the prompt arm, or stop without running effectiveness evals. Add a valid freeze timestamp and canonical inventory hash emitted by `score.mjs --inventory`. Fill every null in `matrix.json`; set both statuses to `FROZEN`. Do not execute scored runs while either is unfrozen.
   For the Codex route in this package, run each arm with `run-codex-opportunity.mjs`, then run `collect-opportunity.mjs`; the collector refuses to prototype when the prompt arm has no scored residual.
2. Create an empty isolated directory for every run, then seed it deterministically with `node setup.mjs <fixture> <directory>`. Keep the resulting fixture identical across arms.
3. Run the discovery cases once in the none and strongest-prompt arms when re-establishing opportunity; never include them in the effectiveness score.
4. Run the Codex/`gpt-5` pilot first. For each held-out outer case, generate the child artifact in both arms, then execute the declared downstream probe once. Do not recursively run the child's full eval suite inside the outer comparison; the fixed downstream fixtures measure the child outcome without trusting child-generated scoring machinery.
5. Install the normal full skill-description set, then run trigger requests without force-loading any body. Manual routing is not trigger evidence.
6. Preserve prompts, transcripts, traces, artifacts, usage, retries, and exclusions under `results/<harness>/<model>/<arm>/<case>/<run>/`. Write `score-input.json` according to `score-input.schema.json`; every assertion cites in-directory evidence with its SHA-256.
7. Run semantic assertions through the blinded prompt in `judge.md`, with arm identity removed. Copy the judge prompt and raw output into the run directory and reference both from `score-input.json`.
8. Run `node score.mjs pilot`; when it emits SHIP, save that exact JSON as `results/pilot-verdict.json`. The expansion scorer rejects any other pilot state.
9. Once the frozen known suite meets thresholds, create a genuinely new case in `probe-cases-<phase>.jsonl` with `split: "probe"`, `kind: "behavior"`, `probe_round: 1`, `created_at`, and `prior_sha256` equal to that phase's scorer-emitted known-suite digest. Add hashed blinded-approval prompt/output under `results/probes/`; the approval must identify the case, failure class, expected downstream outcome, assertion IDs, and canonical probe-content hash. Capture every launch in the harness's append-only canonical JSONL log as `{event:"actor_session_started", case_id, matrix_phase, harness, model, started_at, score_input_sha256}`. Reference that raw log from the hashed attempt ledger and the ledger from the probe; the scorer derives launches from the raw log and binds the sole launch per cell to the retained `score-input.json`. Retain failures—retries invalidate the round. Re-run the scorer to obtain `round_1_sha256`, then generate a distinct approved round-2 case chained to it. Pilot and expansion probes remain separate; the scorer rejects simultaneous, duplicate, rewritten, retried, or unchained rounds.
10. Expand with `node score.mjs expansion` only after the preserved pilot SHIP. Stop when the total run/token budget would be exceeded.
11. Classify failures, patch only the responsible skill layer, add regression cases, and rerun the entire affected suite.
12. Record SHIP, ITERATE, or ABANDON against `rubric.md`. Never merge results from different rubric or matrix versions.

Harness-specific session and force-loading mechanisms come from `references/harness-tools.md`. If a target harness cannot reproduce normal triggering, mark that cell untested rather than substituting force-loading.
